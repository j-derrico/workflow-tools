20 '''  

21 This base class provides the interface to methods used to read in 

22 a user-provided YAML configuration file, convert it to the required 

23 config file, fix files, and namelist for a forecast. Subsequent 

24 methods will be used to stage the input files and run the forecast.  

25  

26 Attributes 

27 ---------- 

28 config_path : Path 

29 The file path to the configuration file to be parsed. 

30 

31 Methods 

32 ------- 

33 requirements()  

34 Recursively parse config and platform files to determine and 

35 fill in any dependencies. 

36 

37 resources() 

38 Determine necessary task objects and fill in resource requirements of each. 

39 Returns a Config object containing the HPC resources needed. 

40 

41 validate() 

42 Validates the objects generated by the driver from the provided 

43 config and platform files. If the --dry_run flag is provided, complete all 

44 stages through validation and print results without running the forecast. 

45 

46 

47 output() 

48 Holds the knowledge for how to modify a list of output files and 

49 stages them in the working directory. 

50 

51 job_card() 

52 Turns the resources config object into a batch card for the  

53 configured Task. Can be called from the command line on the front end  

54 node to show the user what the job card would have looked like. 

55  

56 Notes 

57 ------- 

58 Several functions such as create_model_config() and run() are relegated to 

59 specific forecast drivers. This is because they are unique to the app being 

60 run and will appropriately parse subsequent stages of the workflow.
